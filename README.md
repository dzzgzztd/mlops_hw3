## Описание сценариев НТ

### Сценарий 1. Проверка доступности сервиса
В данном сценарии выполняются запросы к служебному endpoint'у:
- `GET /api/v1/health`

Цель сценария — убедиться в доступности и стабильности API при многократных обращениях.

### Сценарий 2. Сценарий чтения данных
В данном сценарии выполняются запросы:
- `GET /api/v1/models`
- `GET /api/v1/datasets`

Цель сценария — проверить поведение сервиса при типичных операциях чтения данных.

### Сценарий 3. Сценарий инференса
В данном сценарии выполняются запросы:
- `POST /api/v1/models/{model_id}/predict`

Перед запуском сценария предварительно обучается модель и используется её `model_id`.

Цель сценария — измерить производительность операций инференса и получить время ответа модели.

### Сценарий 4. Комбинированная нагрузка
В сценарии смешиваются запросы:
- `GET /api/v1/health`
- `GET /api/v1/models`
- `GET /api/v1/datasets`
- `POST /api/v1/models/{model_id}/predict`

Цель сценария — оценить поведение сервиса в условиях смешанной нагрузки.

## Отчет по НТ

В ходе выполнения задания в приложение были добавлены:
- endpoint `/metrics` для экспорта метрик;
- инструменты для сбора HTTP-метрик;
- пользовательская метрика времени инференса модели;
- VictoriaMetrics для хранения метрик;
- Grafana для визуализации метрик;
- Locust для проведения нагрузочного тестирования.
- Частично добавлен Loki, довести до рабочего состояния не получилось :(

### 1. Собираемые метрики
На дашборде Grafana визуализировались следующие показатели:
- RPS по endpoint'ам;
- latency p50 / p95 / p99;
- error rate;
- время инференса модели;
- RPS операций инференса.

### 2. Параметры тестирования
Для нагрузочного тестирования использовался инструмент Locust.

Параметры запуска:
- количество пользователей: 20–50;
- скорость наращивания нагрузки: 2–5 пользователей в секунду;
- длительность теста: 3–5 минут.

### 3. Результаты
По результатам тестирования были получены и визуализированы следующие характеристики:
- сервис успешно публикует метрики через endpoint `/metrics`;
- VictoriaMetrics корректно собирает и хранит метрики;
- Grafana отображает нагрузочные показатели в реальном времени;
- под нагрузкой наблюдаются значения RPS по endpoint'ам;
- latency p50 / p95 / p99 остаются в контролируемых пределах;
- error rate при корректной работе сценариев равен 0%;
- метрика времени инференса позволяет отдельно оценивать производительность предсказаний модели.

## 4. Выводы
По проделанной работе: В сервис был добавлен endpoint /metrics через Prometheus Instrumentator. Метрики собираются vmagent и сохраняются в VictoriaMetrics. Для визуализации использована Grafana, где настроен дашборд с RPS, latency, error rate и метриками инференса. Нагрузочное тестирование выполнено в Locust, сценарии и отчет приложены в репозиторий.

По метрикам: Сбор метрик происходит корректно, как и отображение в дашборде Grafana. Сервис демонстрирует стабильную работу и низкие задержки при нагрузочном тестировнии, однако HTTP Latency p99 довольно высокий, что свидетельствует о том, что редкие запросы выполняются значительно дольше остальных, может потребоваться оптимизация. Инференс происходит стабильно с незначительными колебаниями.
